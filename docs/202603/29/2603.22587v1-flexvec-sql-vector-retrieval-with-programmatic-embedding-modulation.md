## 摘要
随着 AI 智能体成为检索 API 的主要消费者，有机会向调用者暴露更多的检索流水线。flexvec 是一个检索内核，它将嵌入矩阵和评分数组暴露为可编程表面，允许在选择之前对两者进行算术运算。我们将查询时在该表面上组合操作称为程序化嵌入调制（PEM）。本文描述了一组此类操作，并通过一个支持可组合查询原语的查询物化器将其集成到 SQL 接口中。在包含 240,000 个数据块的生产语料库上，在不使用近似索引的情况下，三种组合调制在台式机 CPU 上端到端执行仅需 19 毫秒。在一百万个数据块的规模下，相同的操作执行时间为 82 毫秒。

## Abstract
As AI agents become the primary consumers of retrieval APIs, there is an opportunity to expose more of the retrieval pipeline to the caller. flexvec is a retrieval kernel that exposes the embedding matrix and score array as a programmable surface, allowing arithmetic operations on both before selection. We refer to composing operations on this surface at query time as Programmatic Embedding Modulation (PEM). This paper describes a set of such operations and integrates them into a SQL interface via a query materializer that facilitates composable query primitives. On a production corpus of 240,000 chunks, three composed modulations execute in 19 ms end-to-end on a desktop CPU without approximate indexing. At one million chunks, the same operations execute in 82 ms.

#### 1. 核心问题与研究动机
随着 AI 智能体（AI Agents）逐渐取代人类成为检索 API 的主要使用者，传统的“黑盒式”检索接口（固定端点、特定 DSL、固定结果格式）限制了智能体的灵活性。
*   **研究动机**：AI 智能体具备直接编写 SQL 的能力，且 SQL 在训练数据中分布广泛。
*   **核心问题**：现有的向量数据库通常不向调用者暴露嵌入矩阵或评分数组，导致无法在查询时进行精细的数学干预。
*   **目标**：开发一个名为 `flexvec` 的检索内核，将向量检索过程转化为一个可编程表面，允许智能体通过 SQL 组合复杂的算术操作（如抑制特定主题、增加多样性、时间衰减等）。

#### 2. 方法论：程序化嵌入调制（PEM）
`flexvec` 的核心思想是**程序化嵌入调制（Programmatic Embedding Modulation, PEM）**，它通过一个“查询物化器（Query Materializer）”将检索过程分为三个阶段：

*   **阶段 1：SQL 预过滤（Pre-filter）**：利用标准 SQL 子查询（如按项目、时间、类型过滤）缩小候选集。这解决了传统向量索引在处理高选择性过滤器时的“池枯竭”问题。
*   **阶段 2：numpy 调制（Modulation）**：在内存中的嵌入矩阵上执行暴力点积计算基础分数，随后应用一系列可组合的算术算子：
    *   `suppress:X`：减去与特定概念 X 的方向相似度（负向查询扩展）。
    *   `decay:N`：根据时间戳进行倒数衰减（偏好近期内容）。
    *   `centroid:ids`：将查询向量移向已知相关示例的质心。
    *   `from:A to:B`：沿特定轨迹（如从原型到生产）调整评分。
    *   `diverse`：使用最大边际相关性（MMR）算法迭代选择，提升结果多样性。
*   **阶段 3：SQL 组合（Compose）**：将评分后的 Top-K 结果作为临时表物化，再通过 SQL JOIN 操作关联元数据，进行最终的排序、分组或重排序。

*   **性能卓越**：在 24 万数据量下，组合调制仅需 19ms；在 100 万数据量下仅需 82ms，证明了暴力检索在百万级规模下的可行性。

*   **透明度**：将检索过程从“黑盒”变为“白盒”，允许开发者或智能体根据上下文动态调整检索策略。

*   **实现方式**：SQL 物化器基于正则表达式重写而非完整解析，可能在处理极复杂的嵌套 SQL 时存在鲁棒性风险。
